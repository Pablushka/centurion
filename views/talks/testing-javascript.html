Testing en JavaScript con Mocha y Chai

Una introducción al testing de aplicaciones web basadas en JavaScript con Mocha y Chai implementando BDD

¿Testing? ¿Para qué?

Reduce los bugs

TDD reduce la densidad de bugs en producción entre un 40-80%.

Fuente: medium.com/javascript-scene/863d5beb3ce9

Mejora el diseño y arquitectura

Te ayuda a modularizar, independizar, enfocar, te obliga a desacoplar y además de ser una documentación viva.

Organización y comunicación

Facilita la organización del trabajo y mejora la comunicación entre miembros del equipo de desarrollo.

Calidad

La confiabilidad y la calidad del producto final es mucho más alta cuando se testea el software

¿Cuándo hacerlo?

En el proyecto

Tamaño, complejidad, calendario, datos, entre otros.

En el equipo

Número de miembros, habilidad, entre otros.

Costos y tiempos

Testear cuesta y requiere tiempo, pero los resultados lo ameritan.

¿Qué es software testing?

Software Testing

Es el proceso de investigar el proyecto con el objetivo de determinar la calidad del software.

Fuente: es.wikipedia.org/wiki/Pruebas_de_software

Métodos

  • Estático vs dinámico: sin ejecutar y ejecutando el software, como linting y demos respectivamente
  • Modelo de caja: white-box, en donde se conoce el código fuente y black-box donde el mismo se desconoce

Niveles

  • Unit testing
  • Integration testing
  • System testing

Unit testing

Unidades o componentes individuales.

Integration testing

Unidades o componentes individuales son combinados y testeados como grupo.

System testing

El software completo o integrado es testeado.

Tipos

  • Funcionales, no funcionales: rasgos técnicos o no técnicos
  • Compatibilidad: si el software funciona bien para diferentes plataformas
  • Rendimiento: si el tiempo de carga o velocidad de respuesta es adecuado
  • Entre muchos otros

Severa panacea ¿cómo testeo?

Automatización

Siempre se testea, sea el proceso manual o automatizado. Aunque es necesario hacer parte del proceso manual, la idea es automatizar lo mayor posible.

TDD y BDD

Test-Driven Development y Behaviour-Driven Development, hace uso de repetir el siguiente ciclo:

  • Crea el test inicialmente fallido
  • Escribe el código para pasar el test
  • Mejora la versión del código

Herramientas

  • Test runner: se encarga de ejecutar de manera organizada y controlada los tests
  • Assertion library: ayuda en la validación y verificación las unidades de testeo
  • Debugger: para encontrar las fuentes de los errores

Test Runner: MochaJS

Nos permite crear "test suites" en Node.js y en el navegador. Corre serialmente tests síncronos y asíncronos, generando reportes y mapeando errores de manera flexible.

Documentación: mochajs.org

Assertion Library: ChaiJS

Crea asertaciones usando los estilos TDD y BDD en Node.js y en el navegador, integrandose bien con cualquier test runner.

Documentación: chaijs.com


$ cd /my/project/folder/
$ npm install chai

src/sum.test.js


const chai = require('chai');
const sum = require('./sum');
const { expect } = chai;

describe('Sum', function () {

  it('Sumar dos números positivos', function () {
    const actual = sum(5, 4);
    const expected = 9;
    expect(actual).to.equal(expected);
  });

  it('Sino recibe parametros retornar 0', function () {
    const actual = sum();
    const expected = 0;
    expect(actual).to.equal(expected);
  });
});

src/sum.js

  return a + b;
};

src/sum.js

  return a + b;
